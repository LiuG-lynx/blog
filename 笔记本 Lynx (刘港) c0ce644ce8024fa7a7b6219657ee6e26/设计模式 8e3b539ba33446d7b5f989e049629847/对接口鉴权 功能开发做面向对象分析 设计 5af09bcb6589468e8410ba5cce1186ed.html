对接口鉴权 功能开发做面向对象分析/设计

案例分析

HTTP 协议暴露接口

统一的 接口鉴权,没有认证调用接口会被拒绝或踢出系统

第一轮基础分析

用户名+ 密码 每次请求都带着 账号密码来和 服务端比对

第二轮分析优化

明文传输密码 不安全

加密密码 一样可以被 重放攻击

调用方 将请求接口+ 账号 + 密码 加密 token 与服务端 token 进行对比 重放攻击

第三轮分析优化

引用时间戳 随机变量

时间戳 + 请求接口+ 账号 + 密码 超时一分钟则判定过期

第四轮分析优化

一分钟内还是有重放攻击 <权衡安全 开发成本 系统性能 > 此方式算比较合理

留有扩展点 可 切换 存储方式 sql nosql zookeeper redis 本地配置等等

确定需求

  1. 时间戳 + 请求接口+ 账号 + 密码 通过加密算法生成 token 发送到 服务端
  1. 服务端解密算法 解析 时间戳, 账号 ,密码,请求接口
  1. 服务端 验证 时间戳是否符合 token 时间窗口内
  1. 服务端验证 token 没有失效 从 存储中拿到 存储的密码 通过同样的 token 生成算法 生成另外一个 token 进行比对 完成正确鉴权

面向对象设计

主要职责 包含以下几个部分

  1. 划分职责进而识别出有哪些类
  1. 定义类属性及方法
  1. 定义 类与类之间的交互关系
  1. 将类组装起来并提供执行入口

1. 划分职责进而识别出有哪些类

根据4点需求所得到的功能点

  1. 把 url ,uid,密码.时间戳 拼接为一个字符串
  1. 对字符串通过加密算法加密生成 token
  1. 将 token ,uid 时间戳 拼接到 url 中, 形成新的 url
  1. 解析 url 得到 token uid 时间戳
  1. 从存储中 取出 token uid 时间戳
  1. 根据 时间戳判断token 是否 过期
  1. 验证 两个 token 是否匹配

由此 可粗略获得三个核心类

AuthToken Url CredentialStorage

AuthToken 负责1,2 ,6,7 跟 token 有关

Url 3,4 处理 url 拼接解析

CredentialStorage 5 操作 uid 密码 存储读取
